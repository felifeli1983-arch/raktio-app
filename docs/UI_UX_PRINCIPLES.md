# UI_UX_PRINCIPLES.md

## Purpose

This document defines the visual, interaction, and product experience principles of Raktio.

Raktio must not look like:
- a generic SaaS dashboard
- a toy AI tool
- a research-lab prototype
- a chaotic data terminal
- a fake social clone

It must look like:
- a premium simulation intelligence platform
- an enterprise-grade analytical product
- a live social world observation console
- a tool users trust for serious decisions

---

## Core experience goal

Raktio should make the user feel:

- “I am observing a living synthetic world”
- “I can understand what is happening”
- “I can control what I inspect”
- “This is serious enough to support real decisions”
- “This is richer than asking one LLM a question”

The product must balance:
- realism
- legibility
- depth
- emotional engagement
- enterprise trust

---

## Foundational UX principles

# 1. Simulation-first, not dashboard-first

The product center is not a dashboard of cards.  
It is the **Simulation Canvas** and the workflows around it.

Dashboards and overview pages exist to:
- orient
- prioritize
- route attention
- surface important changes

They do not define the product.

---

# 2. Show the system, not just the content

Raktio must not only display:
- posts
- comments
- pretty charts

It must show:
- relationships
- spread
- divergence
- belief shifts
- geography
- platform differences
- causal dynamics

The user must feel they are seeing a **system**, not a stack of outputs.

---

# 3. Dense but readable

Raktio must be information-dense.  
But density must come from:
- strong hierarchy
- consistent spacing
- disciplined visual grouping
- progressive disclosure
- good defaults

Not from:
- cramming widgets everywhere
- giant empty spaces
- decorative clutter
- raw logs dumped as UI

---

# 4. One world, many lenses

The same simulation should be observable through multiple lenses:
- Feed
- Network
- Timeline
- Geo
- Segments
- Compare

These are not separate products.  
They are different viewpoints over the same living simulation.

The UI should reinforce this unity.

---

# 5. Dopamine through meaning, not gimmicks

Raktio should be compelling because users can see:
- a cascade starting
- a faction splitting
- a geography diverging
- a belief shifting
- a comparison winning

Not because of:
- random motion
- sci-fi glow overload
- theatrical effects
- gratuitous transitions

---

## Visual positioning

Raktio should feel closer to:
- premium data products
- intelligence platforms
- advanced product analytics tools
- market research interfaces
- social listening war rooms

It should not feel closer to:
- crypto dashboards
- hacker terminals
- game HUDs
- academic prototypes
- social media toy mockups

---

## Visual style principles

# 1. Clean enterprise base

The visual base should be:
- clean
- structured
- modern
- premium
- slightly technical
- not sterile

This means:
- strong layout grid
- well-controlled typography
- limited, meaningful color usage
- clear contrast
- disciplined modules

---

# 2. Social layer on top of enterprise layer

Raktio has a dual nature:
- enterprise analysis product
- social simulation product

The UI must reflect both.

### Enterprise layer
- structure
- navigation
- cards
- metrics
- tables
- governance pages

### Social layer
- feed cards
- avatars
- interaction indicators
- thread behavior
- platform nuance
- visible social energy

The UI must merge these two worlds cleanly.

---

# 3. Information hierarchy must be explicit

Every page should clearly communicate:
- primary object of attention
- secondary context
- live status
- actionable next step

Users must never ask:
- “what am I supposed to look at here?”
- “why is this on the screen?”
- “what matters most?”

---

# 4. Visual consistency across modes

Feed mode, Network mode, Timeline mode, Geo mode, and Compare mode should feel like:
- one platform
- one design language
- one system

Not like five different tools stitched together.

---

## Layout rules

# 1. Stable shells, dynamic centers

Navigation, major chrome, and high-level structure should remain stable.

The central working area can change by mode:
- feed
- graph
- timeline
- geo
- compare

This gives users:
- orientation
- confidence
- rhythm
- lower cognitive overhead

---

# 2. Side rails must earn their space

Left rails and right rails should not exist just because SaaS apps often have them.

### Left rail should hold
- filters
- scope controls
- saved views
- analytical lenses

### Right rail should hold
- live metrics
- alerts
- summary intelligence
- current world state

If the content is weak, collapse or simplify the rail.

---

# 3. Main area should always carry the core story

The central area must always show:
- what is happening
- where it is happening
- how it is evolving

This is especially important in Simulation Canvas and Reports.

---

# 4. Empty states must feel guided, not dead

When there is no data yet, the screen must still communicate:
- what will appear here
- why it matters
- what the user should do next

No barren enterprise emptiness.

---

## Typography principles

# 1. Serious, readable, product-grade typography

Typography should communicate:
- clarity
- analysis
- trust
- modernity

Not:
- playfulness
- sci-fi fantasy
- decorative branding excess

### Suggested role split
- strong headings
- crisp section labels
- compact metadata
- readable body text
- consistent numeric typography for metrics

---

# 2. Numbers and labels need strong discipline

Metrics must be:
- aligned
- easy to scan
- visually grouped
- consistently formatted

This is especially important for:
- live simulation stats
- compare deltas
- scorecards
- billing/credit pages
- admin/cost pages

---

## Color principles

# 1. Color must mean something

Color should communicate:
- sentiment
- risk
- status
- platform identity
- category grouping
- alert intensity

It should not be used randomly.

---

# 2. Sentiment color system

There should be a clear sentiment system, for example:
- positive
- neutral
- negative

But these colors must be:
- visible
- restrained
- consistent across feed, charts, geo, and reports

---

# 3. Platform identities should be clear but not overwhelming

Each platform can have a recognizable visual identity layer, but:
- do not turn the whole product into a rainbow of brands
- keep the base system coherent
- let platform identity appear in badges, content treatment, and mode-specific rendering details

---

# 4. Alert color must be distinct from sentiment

Risk/alert states are not the same as sentiment.

An alert system should clearly distinguish:
- soft signals
- warning signals
- urgent anomalies
- critical divergence

---

## Card and module design rules

# 1. Cards must have a purpose

Do not fill pages with generic cards.

Each card/module should answer:
- what does this tell me?
- why is it here?
- what can I do next?

---

# 2. Cards should support scanning

Good cards should expose:
- title
- key signal
- one supporting detail
- one action or drill-down path

Avoid giant blocks of vague text.

---

# 3. Social cards must be augmented, not overloaded

In the feed:
- posts should feel like posts
- comments should feel like comments
- threads should feel natural

But the augmentation layer should add:
- sentiment markers
- stance badges
- platform indicators
- influence hints
- turning-point labels

This augmentation must remain elegant.

---

## Interaction principles

# 1. Drill-down everywhere meaningful

Users should be able to go from:
- high-level signal
to
- supporting detail
without friction

Examples:
- report finding -> evidence
- geo hotspot -> local feed
- agent node -> agent profile
- compare delta -> timeline divergence

---

# 2. Progressive disclosure over overload

Default views should show:
- the most important information

Deeper views should unlock:
- more detail
- evidence
- diagnostics
- internals

This is how you keep the product rich but not overwhelming.

---

# 3. Controls should feel investigative

Filters and controls should feel like analytical tools, not like boring enterprise forms.

Examples:
- focus this cluster
- isolate this geography
- only show controversial content
- show belief shifts
- interview this agent
- compare before vs after turning point

---

# 4. User actions must create momentum

Good actions in Raktio:
- interview agent
- clone and modify
- create variant
- save audience
- compare simulations
- bookmark turning point
- ask the crowd

These actions deepen engagement and retention.

---

## Simulation Canvas-specific principles

# 1. The world must feel alive quickly

Users should enter the canvas as soon as meaningful runtime activity exists.

Even if the simulation is not complete, they should feel:
- movement
- emergence
- consequence

---

# 2. Feed must be emotionally legible

The feed should show:
- tension
- support
- skepticism
- mockery
- momentum
- disagreement

Without becoming cartoonish.

---

# 3. Graph mode must be interpretable

The network must not become:
- an unreadable hairball
- a decorative graph wallpaper

It should emphasize:
- clusters
- bridges
- amplifiers
- faction shifts
- causal chains

---

# 4. Geo mode must never be static

Single-country runs must still feel dynamic through drill-down.

Geo should support:
- discovery
- divergence
- local nuance
- not just “map as decoration”

---

# 5. Compare must feel powerful

Compare mode should make improvement obvious.

The user should feel:
- “I changed this one thing, and now I can see the difference.”

That feeling is central to the product loop.

---

## Reports-specific principles

# 1. Reports must feel premium and inspectable

Reports should combine:
- structure
- narrative clarity
- evidence
- visual clarity
- interaction

They must not be:
- a giant wall of prose
- a chart dump
- generic LLM explanation

---

# 2. Reports must lead to action

Every major report section should make it easier to:
- change something
- rerun something
- compare something
- save something

Reports are not the end of the journey.

---

## Audience and Agent UX principles

# 1. Agents must feel like inhabitants, not records

Agent profiles should communicate:
- continuity
- recognizability
- memory
- specificity
- realism

### Avoid
- “database row” feel
- throwaway identity
- flat profile pages

---

# 2. Audience creation must feel like population design

Audience Studio should feel like:
- building/selecting a synthetic population
not
- filling out an HR-style filter form

The language and controls matter.

---

## Billing and commercial UX principles

# 1. Credit consumption must be understandable

Users should always understand:
- why a simulation costs what it costs
- what changes the cost
- what upgrades unlock
- what extra credits do

No hidden-feeling credit drain.

---

# 2. Pricing must feel fair but premium

The UI should communicate:
- flexibility
- scalability
- clarity
- control

Not:
- nickel-and-dime psychology
- confusing credit math
- arbitrary restrictions

---

## Admin and enterprise UX principles

# 1. Admin pages should be powerful but calmer

Admin areas should feel:
- structured
- operational
- trustworthy
- controllable

Not:
- noisy
- experimental
- overly visual

---

# 2. Enterprise controls must feel intentional

Enterprise features should feel like:
- governance
- reliability
- scalability
- managed control

Not:
- random “advanced settings” piles

---

## What Raktio must avoid visually

Raktio must avoid:

- giant empty whitespace with little signal
- visual chaos caused by too many competing widgets
- too-dark “hacker terminal” theatrics
- fake-futuristic sci-fi styling
- cheap social clone aesthetics
- cards for the sake of cards
- dashboards with no hierarchy
- raw technical logs presented as product UI
- overdependence on tables where mixed views would be better
- inconsistent design language across modules

---

## Ideal emotional tone

Raktio should feel:

- intelligent
- sharp
- controlled
- modern
- dynamic
- serious
- slightly addictive in a good way
- confident but not arrogant

It should not feel:
- childish
- gimmicky
- over-branded
- fake-“AI magical”
- overly academic

---

## Design consistency requirements

Across all pages, maintain consistency in:
- spacing system
- typography scale
- card behavior
- badge treatment
- filter style
- chart style
- avatar treatment
- status chips
- alert treatment
- panel structure
- drill-down behavior

This is crucial because the app surface area is large.

---

## Final rule for Claude

**Design Raktio as a premium simulation intelligence product with a clean enterprise foundation and a live social-world layer on top. The experience must be rich, legible, dynamic, and trustworthy. Never let it slip into generic SaaS boredom, research-tool awkwardness, or theatrical sci-fi noise.**
